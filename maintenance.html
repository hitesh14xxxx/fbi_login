Optimization Discussion: Refactoring `script.js` Login Logic

FM
FiveM. Dev
Posted 3 days ago

I've been reviewing the logic in the login section of **`script.js`**. The current approach to authentication—where it attempts a server login *and then* falls back to a synchronous $\text{localStorage}$ check—is messy. It leads to duplicate success logic and violates the principle of separation of concerns.

Specifically, this section needs cleanup: 

attemptServerLogin().then(result => {
  // ... remove loading state ...

  if (result.ok) {
    // SUCCESS LOGIC DUPLICATION 1
    // ... show success toast ...
    setTimeout(() => { window.location.href = 'maintenance.html'; }, 400);
  } else {
    // ... FALLBACK LOGIC ...
    let ok = false;
    // ... localStorage check ...

    if (ok) {
      // SUCCESS LOGIC DUPLICATION 2
      // ... show success toast ...
      setTimeout(() => { window.location.href = 'maintenance.html'; }, 400);
    } else {
      // ... show invalid credentials ...
    }
  }
});

How can we wrap the success action into a single, clean function to avoid $\text{DRY}$ violation, even with the dual auth path?

Good catch. The easiest fix is to move the redirect/toast logic out of the $\text{Promise}$ chain entirely and call it from a single place if *either* the server or $\text{localStorage}$ check passes. Use a single boolean `isSuccess` after both checks run.

Even better: since `attemptServerLogin` is an $\text{async}$ function, just change the fallback logic to check $\text{localStorage}$ *before* the server call, but only if the server call fails. 

const login = async () => {
  const serverResult = await attemptServerLogin();
  if (serverResult.ok) return handleSuccess();

  const localStorageOk = checkLocalStorage();
  if (localStorageOk) return handleSuccess();

  handleFailure();
};
